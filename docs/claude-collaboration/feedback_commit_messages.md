커밋 메시지를 작성할 때는 비개발자(예: PM, 운영자, QA)도 한 번 읽고 무엇이 바뀌는지 알 수 있게 쓴다.

**Why:** 사용자가 2026-04-29 명시 — "커밋 메시지는 비개발자도 이해가능하도록 작성해. 앞으로도." 본 프로젝트의 commit 이 운영자/QA 가 보는 채널과 연결되어 있어 *기능 단위로 무엇이 바뀌었는지* 가 더 중요.

**How to apply:**
- 첫 줄 (subject) 에 *사용자 관점* 의 변화를 기술. 함수명·클래스명 사용 금지. 한국어 권장.
  - 좋음: `로그인 만료 시 안내 모달이 뜨도록 수정`
  - 나쁨: `ReplayAuthExpiredError 를 ReplayProxyError 와 분리해 409 분기`
- 본문은 bullet 으로 *변경 내용 + 사용자 영향* 짝을 명시. 코드 아닌 *결과* 를 묘사.
  - 좋음: `재생 도중 인증 만료가 감지되면 빨간 알림창이 뜨고 [재시드] 버튼이 폼을 자동으로 채워줌`
  - 나쁨: `app.js 의 _runPlay 에 409 catch 추가 + reseedPrefill fetch`
- 보안 / 안전 / 데이터 손실 관련 변경은 별도 강조 줄 (`보안:` `주의:`).
- 내부 코드 위치 (파일명/함수명) 는 *bullet 끝의 괄호* 에만, 또는 별도 "기술 메모" 섹션으로 분리.
- conventional-commit prefix (`feat`, `fix`, `docs`, `chore`) 는 사용해도 OK — 사용자가 이미 사용 중. 단 prefix 뒤 주제는 평이한 한국어.
- Phase / 의사결정 코드 (예: `D16`, `P3.7`) 는 중요하면 괄호로 짧게만.

이 규칙은 자동 생성 메시지(예: `Co-Authored-By` 줄, hook 이 박는 것) 에는 적용되지 않음 — 사람이 작성하는 subject + body 에만 적용.
