모든 의사결정과 구현사항을 **상세히 문서화** 한다. 코드만 남기고 끝내지 말 것.

**Why:** 사용자가 2026-05-02 명시적으로 "앞으로도 계속" 적용하라고 지시. 비개발자/이해관계자가 변경 흐름을 사후 추적할 수 있어야 함. CLAUDE.md 의 "Default to writing no comments" 는 이 사용자에 한해 *논리·왜* 차원에서는 무시 — 다만 코드 내 짜잘한 주석은 여전히 자제, **설계 문서 / 결정 로그** 가 정공법.

**How to apply:**
- **비-사소한 변경 (3개 파일 이상 또는 동작 변경) 시** : `playwright-allinone/docs/` 아래에 `PLAN_*.md` 또는 `DECISION_*.md` 형식의 설계 문서를 함께 작성·커밋. 항목별 일목요연 마크다운 (이미 잡혀 있는 톤).
- 문서 구성: `## 배경` / `## 의사결정` (대안 + 채택 사유 + 트레이드오프) / `## 구현 범위` / `## 검증` / `## 후속 작업`.
- 코드 내 inline 주석은 여전히 자제 — "왜" 가 비-자명한 지점만. 설계 문서가 1차 산출물.
- 커밋 메시지는 "## 변경 / ## 배경 / ## 검증" 구조 유지 (`feedback_commit_message_style` 와 일관).
- 사용자가 동의한 경로 / 정한 라벨 / 거절한 대안 등 **결정의 근거** 가 명확히 남아 있어야 향후 다른 conversation 에서도 참조 가능.

**예외/완화:**
- 한두 줄 fix (오타·라벨 수정·번역 등) 은 커밋 메시지만으로 충분 — 별도 문서 불필요.
- "탐색 → 결정 → 진행" 흐름의 채팅 그 자체가 결정 로그 역할이므로 사용자 요청에 따른 단일 단계 작업은 메모리 + 커밋 메시지로 충분. **여러 단계 / 깊은 변경** 일 때만 별도 docs/ 파일.
