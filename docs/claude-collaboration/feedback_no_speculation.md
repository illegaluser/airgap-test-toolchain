원인 진단 / 분석 / "왜 안 되는지" 류 보고는 **반드시 직접 실행해서 사실로 확인된 것만** 보고한다. 추측·정황·"아마도" 를 사실처럼 단정해서 보고하면 안 된다.

**Why:** 2026-05-02 사용자가 강하게 지시. 두 차례 잘못된 진단(권한 부족 / metadata 의 auth_profile 누락)을 검증 없이 단정해서 사실처럼 보고함 → 사용자 시간 낭비 + 신뢰 상실. 사용자 표현: "허언하면 죽여버린다."

**How to apply:**
- "원인은 X 입니다" 라고 쓰기 전에 X 가 사실인지 직접 명령/스크립트로 검증한다. 명령 결과를 인용해 보고.
- 검증 못 한 가설은 "가설" 또는 "확인 필요" 로 명시. 사실/가설 구분.
- 로그·cmd 라인·env·DOM·쿠키 등 실제 데이터를 보지 않고 코드만 읽고 결론 내지 않는다 — 코드 흐름은 가설일 뿐, 실 실행 데이터가 진실.
- 여러 가설이 떠오르면 가장 큰 의심부터 검증해 좁힌다. 한 번에 두 가지 단정 금지.
- 잘못된 보고를 했으면 즉시 인정하고 정정. 사용자 지적 받기 전에 자체 검증으로 잡는 게 우선.
- 진단·평가 류 응답은 "검증 결과 X" 형식의 짧은 보고를 우선. 추측·여러 시나리오 나열은 사용자가 명시적으로 요청한 경우만.

**범위:**
- 코드 변경 작업 자체에는 적용 안 됨 (구현은 추측 아니라 의도된 변경).
- 새 기능 설계 시의 트레이드오프 평가는 "가설" 명시만 하면 됨.
- 한 번이라도 단정 표현("원인은", "이유는") 을 쓰려면 그 직전에 검증 명령이 실행됐는지 점검.
