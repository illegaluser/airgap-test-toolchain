# 경계값 분석 (Boundary Value Analysis) — 테스트 설계 기법

## 개요

경계값 분석 (BVA) 은 **입력 / 출력의 경계 부근에서 결함이 가장 빈번하게 발견된다**는
경험적 사실에 근거한 테스트 설계 기법이다. 동등 분할 (Equivalence Partitioning) 과
함께 가장 보편적인 black-box 테스트 기법.

## 핵심 원칙

각 입력 변수에 대해 다음 5 개 값을 테스트:

1. **최소값 - 1** (`min - 1`) — 무효 영역
2. **최소값** (`min`) — 유효 영역의 경계
3. **정상값** (`normal`) — 유효 영역 내부
4. **최대값** (`max`) — 유효 영역의 경계
5. **최대값 + 1** (`max + 1`) — 무효 영역

## 적용 예시

### 예시 1: 비밀번호 길이 (8 ~ 64자)

| 입력 길이 | 분류 | 기대 결과 |
| --- | --- | --- |
| 7자 | min - 1 | 거절 — "최소 8자" 에러 |
| 8자 | min | 통과 |
| 30자 | normal | 통과 |
| 64자 | max | 통과 |
| 65자 | max + 1 | 거절 — "최대 64자" 에러 |

### 예시 2: 결제 금액 (1 ~ 9,999,999)

| 금액 | 분류 | 기대 |
| --- | --- | --- |
| 0 | min - 1 | 거절 |
| 1 | min | 통과 |
| 50000 | normal | 통과 |
| 9999999 | max | 통과 |
| 10000000 | max + 1 | 거절 |

## 함정 (Pitfalls)

### 1. off-by-one

- 코드가 `if (length > 8)` 인지 `if (length >= 8)` 인지에 따라 8자 결과가 달라짐.
- BVA 가 잡아내는 가장 흔한 결함.

### 2. 다중 변수 동시 경계

- 두 변수 (예: 가로 × 세로) 가 모두 경계일 때 (`min × max`) 의 동작은 경계값
  단독 테스트로 확인 안 됨. **복합 경계 (combined boundary)** 별도 테스트 필요.

### 3. 데이터 타입 경계

- int32 max (2147483647) 같은 시스템 한계도 별도 점검.

## 동등 분할과의 결합

BVA + 동등 분할 (EP) 조합:

1. **EP 로 입력을 유효 / 무효 클래스로 분할**.
2. **각 클래스의 경계에서 BVA 적용**.
3. 클래스 내부 정상값 1 개 + 양 끝 경계값 + 인근 무효값 = 클래스당 ~5 개 케이스.

## 회귀 정책

- 새 기능 추가 시 BVA 케이스를 회귀 셋에 포함.
- 기존 BVA 케이스가 깨지면 즉시 차단 (regression).
- 입력 범위 확장 (예: max 9999999 → 99999999) 시 새 max-1, max+1 추가.

## 자동화 권장 기법

- **fixture HTML 의 input 에 min/max attribute 가 있다면** 그 값을 그대로 BVA 입력으로 채택.
- **API 문서의 params 컬럼** 에 type / range 가 있으면 자동 생성 가능 (예: `{amount:int, min:1, max:9999999}` → BVA 5 케이스 자동 emit).
- 수동 작성 시 문서화: 각 BVA 케이스 옆에 "근거: spec §X.Y" 명시.
