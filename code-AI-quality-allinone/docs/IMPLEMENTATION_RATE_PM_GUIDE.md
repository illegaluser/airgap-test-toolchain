## 한 줄 요약

**요구사항 명세서 1 부 + 빌드 1 회 → 구현율 1 숫자 + 근거 매핑표 + 변화 알림.**

정확도는 학술 검증·교차 검증·사람 정답지 3중 보장으로 70% 이상, 자동 판정이 애매한 4~10% 만 사람에게
위임 → 검수 부담 약 90% 절감.

## 1. 비유로 보면 — *"시험 답안 자동 채점기"*

본 시스템은 다음과 같이 동작합니다:

| 비유 | 본 시스템 |
|---|---|
| 시험지 | **요구사항 명세서 (PRD)** |
| 답안지 | **소스 코드** |
| 채점관 | **AI (사내 구동)** |
| 채점 결과 | "구현됨 / 부분 구현 / 미구현 / 측정 대상 아님 / 사람 검수 필요" 5 단계 |
| 점수표 | **구현율 리포트** (전체 분포 + 영역별 + 변화 추이 + 근거 매핑) |

채점관(AI) 은 시험지의 각 문항을 보고 *"이 문항에 대한 답이 답안지 어디에 있는가"* 를 찾아 매기고,
애매한 문항은 *"사람이 봐야 합니다"* 로 분리해 따로 보냅니다.

## 3. 어떤 원리로 측정하는가 — 4 단계

### 1 단계 · 요구사항을 *단위로* 쪼개기

요구사항 명세서는 보통 길고 복합적입니다. 자동 측정을 위해 *더 이상 쪼갤 수 없는 단위* 로 분해합니다.

| 원본 (자유 문장) | 쪼갠 결과 |
|---|---|
| "관리자가 신규 고객을 등록하고, 정보를 수정·삭제할 수 있어야 한다." | **REQ-001**: 신규 고객 등록<br>**REQ-002**: 고객 정보 수정<br>**REQ-003**: 고객 정보 삭제 |

각 단위에 *ID + 카테고리 + 합격 기준* 을 자동 부착합니다 (예: REQ-001 의 합격 기준 = "POST /customers
엔드포인트 존재 + 필수 필드 누락 시 400 반환").

> *"빠르게 동작해야 한다"* 같은 모호한 문장은 자동 측정 불가 → "측정 대상 아님 (n/a)" 으로 분류해
> 별도 버킷으로 분리합니다 (전체 구현율 분모에서 제외).

### 2 단계 · 각 요구사항에 해당하는 코드를 *세 가지 방법으로 동시에* 찾기

도서관 사서가 책을 찾는 세 가지 방법과 같습니다:

| 방법 | 비유 | 본 시스템에서 |
|---|---|---|
| **의미 검색** | 같은 주제의 책 찾기 | 요구사항의 의미와 비슷한 코드 찾기 ("고객 등록" → register, sign-up, addCustomer 등) |
| **이름 검색** | 책 제목·저자명 색인 | 요구사항 단어가 그대로 등장하는 코드 찾기 |
| **관계 검색** | 인용·참고문헌 네트워크 | 호출관계 그래프 따라가기 (이 함수를 호출하는 함수 / 이 클래스를 상속하는 클래스) |

세 방법으로 찾은 후보들을 종합해 *상위 5~10 개* 코드 조각만 추립니다.

> 셋 다 동시에 쓰는 이유: 도서관에서 *주제별 분류* 만으로는 *제목으로 찾는 책* 을 못 찾고,
> *제목 색인* 만으로는 *비슷한 주제의 책* 을 못 찾기 때문입니다.

### 3 단계 · AI 채점관이 판정

쪼갠 요구사항 + 후보 코드 5~10 개 → AI 가 검토 → 구조화된 응답:

```
요구사항: "관리자가 신규 Owner 등록 가능 (이름·주소·전화 필수)"

판정: ✅ 구현됨
근거 코드: OwnerRestController.addOwner
        (src/main/java/.../OwnerRestController.java:42)
신뢰도: 0.92
판단 이유: POST /owners 엔드포인트가 존재하고, @Valid 어노테이션으로
         필수 필드 검증이 적용되어 있음. 누락 시 400 Bad Request 반환.
```

판정은 *5 가지 결과* 중 하나로 자동 분류됩니다:

| 판정 | 의미 |
|---|---|
| ✅ **구현됨** (implemented) | 요구사항이 코드에 명확하게 구현됨 |
| 🟡 **부분 구현** (partial) | 일부만 구현됨 (예: 등록은 있지만 검증 누락) |
| ❌ **미구현** (not implemented) | 코드에서 찾을 수 없음 |
| ⚪ **측정 대상 아님** (n/a) | 모호한 요구사항이라 자동 측정 불가 |
| 👤 **사람 검수 필요** (needs-review) | 자동 판정이 애매하거나 두 번 판정이 엇갈림 |

### 4 단계 · 교차 검증으로 신뢰도 보강

AI 의 가장 큰 위험은 *"자신 있게 틀린 답"* 입니다. 이를 막기 위해:

1. **같은 AI 가 다른 방식의 질문으로 두 번 판정** (다른 prompt, 같은 모델)
2. **두 판정이 일치하면** → 자동 마감 (자동 점수)
3. **엇갈리면** → "사람 검수 필요" 큐로 이동
4. **자동 마감되어도 신뢰도가 낮으면** (0.85 미만) → 역시 사람에게 위임

> 결과적으로 *AI 자동 판정 96% + 사람 검수 4%* 정도의 분배가 됩니다 (프로젝트 따라 ±5%).

## 4. 왜 이 숫자를 믿을 수 있나 — 4 가지 근거

### 근거 ① · 학술 검증 (30 년 이상 연구된 분야)

요구사항 ↔ 코드 매핑은 *Trace Link Recovery (TLR)* 라는 분야로 1990년대부터 학계에서 다뤄왔습니다.
2024~2026 의 최신 연구 성과:

| 연구 (출처) | 정확도 |
|---|---|
| TraceLLM (2026) — 가장 어려운 벤치마크 | 58.75% |
| ProReFiCIA — 회수율 (놓치지 않는 비율) | 93.3% |
| 자동차 도메인 RAG 기반 TLR | 검증 99% / 회수 85.5% |

본 시스템은 같은 패턴 (Hybrid Retrieve + LLM Judge) 을 따르므로 현실적으로 *70% 이상* 달성 가능.

### 근거 ② · AI 가 *모르는 것을 모른다고 말하게* 함 (Abstention)

AI 가 자신 있게 틀린 답을 내는 것을 막는 4 가지 보정:

| 보정 항목 | 의미 |
|---|---|
| 인용한 코드 위치가 너무 적으면 | 신뢰도 자동 하락 |
| 검색 결과 1위 점수가 낮으면 | 신뢰도 자동 하락 |
| 세 검색 방법 중 하나만 결과가 있으면 | 신뢰도 자동 하락 |
| 두 번 판정이 엇갈리면 | 자동 마감 거부 → 사람에게 위임 |

→ 신뢰도 0.85 미만은 *무조건* 사람 검수 큐로. AI 가 자기 한계를 인지하는 구조.

### 근거 ③ · 사람 검수와의 일치율을 *측정* (Ground Truth)

자동 판정만으로는 신뢰할 수 없으므로, *사람이 직접 작성한 정답지* 와 비교해 정확도를 별도 측정합니다.

- 별도로 사람이 12 개 요구사항에 대한 정답지 작성
- 자동 판정 결과 vs 사람 정답지 → 정확도(precision) / 회수율(recall) 산출
- **1 차 목표 일치율 50% 이상 / 튜닝 후 70% 이상**
- 이 검증을 거친 후에야 운영 단계 진입

→ 이해관계자에게 *"본 측정의 정확도는 70%+ 보장"* 으로 명확히 발화 가능.

### 근거 ④ · 폐쇄망 자급 (외부 의존성 0)

본 시스템은 외부 클라우드 API 를 일체 호출하지 않습니다.

| 항목 | 본 시스템 |
|---|---|
| 코드가 외부로 나가는가 | ❌ 절대 안 나감 (NDA·보안 보장) |
| AI 모델 출처 | 사내 단일 머신에서 자체 구동 |
| 검색엔진·DB 출처 | 모두 사내 자체 구동 |
| 외부 API 호출 | 0 건 |

→ SI 환경의 보안·NDA 요건과 완전 부합.

## 5. 정직한 한계 — 미리 알려야 할 것

PM 이 기대치를 정확히 잡을 수 있도록 *못 하는 것* 도 명확히 알립니다.

| 한계 | 의미 | 대응 |
|---|---|---|
| 모호한 요구사항 ("빠르게 동작해야 함") | 자동 측정 불가 | "측정 대상 아님" 분류 → 분모에서 제외 |
| 1차 측정 정확도 50~70% | 즉시 운영 불가 | 튜닝 1~2 주 후 70%+ 달성하면 운영 시작 |
| PRD 형식 다양성 (Word·Confluence·Markdown 혼재) | 자체 파서 추가 작업 | 사전에 형식 표준화 권장 |
| 회귀 감지의 *원인* 자동 식별 | "왜 떨어졌는지" 는 사람 판단 | 어떤 요구사항이 떨어졌는지는 정확히 보여줌 |
| 코드 외 산출물 (UI 디자인·DB 스키마 등) | 측정 범위 밖 | 별도 검증 프로세스 권장 |

## 6. PM 이 실전에서 어떻게 쓰는가

| 상황 | 무엇을 보면 되나 | 어떤 결정을 내리나 |
|---|---|---|
| **스프린트 종료** | 영역별 분포 (auth 80% · payment 50% ⚠ · ops 90%) | 가장 약한 영역을 다음 스프린트 우선순위로 |
| **릴리스 게이트 판정** | 전체 구현율 vs 목표 (예: 73% vs 80%) | GO / NOGO 결정 |
| **이해관계자 보고** | 근거 매핑표 (요구사항 → 코드 위치) | "REQ-X 는 OwnerRestController.addOwner 에서 구현됨" 근거 제시 |
| **회귀 감지** | 전월 대비 변화 + 어느 요구사항이 떨어졌는지 | 해당 영역 코드 변경 이력 추적 |
| **사람 검수 효율** | 사람 검수 큐 (보통 4~10%) | 전체가 아닌 *애매한 항목만* 검수 → 부담 90% 절감 |
| **계약 협상·납품 보고** | 학술 근거 + ground truth 검증 결과 | "정확도 70%+ 검증된 측정 시스템" 으로 신뢰성 확보 |

## 7. 자주 묻는 질문 (FAQ)

### Q1. 이 시스템이 *사람 검수를 완전히 대체* 하는가?

**아니요.** 사람 검수를 *보강* 합니다. 자동 판정이 명확한 90~96% 는 자동 마감하고, 애매한
4~10% 만 사람에게 보내 *검수 부담을 90% 절감* 하는 구조입니다. 자동 판정 결과도 PM 이 언제든
근거 매핑표로 spot check 할 수 있습니다.

### Q2. 정확도가 70% 라면, 30% 는 틀린 건가?

**그렇지 않습니다.** 30% 는 다음 중 하나로 분류됩니다:
- "측정 대상 아님" (모호한 요구사항)
- "사람 검수 필요" (자동 판정이 애매함)
- 그리고 일부가 *실제 오판*

진짜 오판 비율은 보통 5~15% 정도이며, 이는 *학술 벤치마크 수준* 입니다. 또한 보수적 임계값으로
*확신 없는 판정은 자동으로 사람에게 위임* 하므로 silent failure 위험이 낮습니다.

### Q3. AI 가 코드를 *볼 수 있나*? 외부에 정보가 나가지 않는가?

**AI 모델 자체가 사내 머신에 설치되어 있습니다.** 외부 클라우드 (OpenAI / Claude / Gemini 등) 호출
0 건. 코드·요구사항·판정 결과 모두 폐쇄망 안에서 완결됩니다. SI 환경의 NDA·보안 요건과 부합.

### Q4. 얼마나 자주 측정되나?

- **빌드 트리거 시** 자동 실행 (Jenkins 파이프라인)
- 또는 **야간 cron** (예: 매월 1일 02:00) 으로 정기 측정
- 100 개 요구사항 기준 약 **20~50 분** 소요 (모델·머신에 따라)

### Q5. PRD 가 자주 바뀌면 어떻게 되나?

- 매 빌드마다 *최신 PRD* 로 재측정
- 전월 대비 변화를 자동 감지 → drift 알림
- 단, 요구사항 ID 가 안정적으로 유지되어야 추적 가능 (REQ-001 같은 식별자)

### Q6. 이 시스템을 도입하면 *무엇이 좋아지나*?

| 영역 | 도입 전 | 도입 후 |
|---|---|---|
| 구현율 측정 | 사람이 수기 추적 (1주~2주, 부정확) | 빌드마다 자동 (몇 십 분) |
| 회귀 감지 | 사후 발견 (다음 sprint 회의 등) | 빌드 직후 자동 알림 |
| 사람 검수 부담 | 100% 검수 | 4~10% 만 검수 |
| 이해관계자 보고 | 주관적 진척도 보고 | 객관적 숫자 + 근거 매핑 |
| 계약·납품 근거 | 정성적 | 학술 검증된 정량 측정 |

### Q7. 첫 측정 결과가 50% 라면 *프로젝트가 50% 진행* 된 건가?

**그 해석은 부정확합니다.** 이 숫자는:
- *PRD 의 atomic 요구사항 중* 코드 구현 증거가 발견된 비율
- *프로젝트 일정 진척도* 와는 다른 개념
- 영역별 편차가 더 중요 (어떤 영역이 약한지)
- 절대값보다 *변화 추이* (drift) 가 의사결정에 유용

---

## 8. 한 페이지 요약 (반복)

| 질문 | 답변 |
|---|---|
| **무엇을** 측정하나 | PRD 의 atomic 요구사항이 코드에 얼마나 구현되었는가 |
| **어떻게** 측정하나 | 요구사항 분해 → 세 가지 방법 검색 → AI 판정 → 교차 검증 |
| **얼마나** 정확한가 | 학술 검증된 패턴 + 사람 정답지 검증으로 **70%+ 보장** |
| **무엇을** 받게 되나 | 한 페이지 리포트 (구현율 + 영역별 + 변화 + 근거 매핑) |
| **언제** 측정되나 | 빌드마다 또는 야간 cron (월 1회 등) |
| **사람은** 무엇을 하나 | 4~10% 의 *애매한 항목* 만 검수 → 부담 90% 절감 |
| **외부에** 정보가 나가나 | ❌ 절대 안 나감 (폐쇄망 자급) |
| **이해관계자에게** 무엇을 말할 수 있나 | "학술·교차·정답지 3중 검증된 70%+ 정확도 측정 시스템" |

## 부록 · 용어 정리

| 용어 | 설명 |
|---|---|
| **PRD** (Product Requirements Document) | 제품 요구사항 명세서 |
| **Atomic 요구사항** | 더 이상 쪼갤 수 없는 단일 요구 단위 (예: "신규 고객 등록") |
| **Acceptance criteria** | 합격 기준 (예: "POST /customers 엔드포인트 존재") |
| **Trace matrix** | 요구사항 → 코드 위치 매핑표 (drill-down 가능) |
| **Drift** | 시간에 따른 변화 (전월 대비 -10pp 같은 회귀 알림) |
| **Ground truth** | 사람이 직접 작성한 정답지 (정확도 측정용) |
| **Abstention** | AI 가 *"잘 모르겠다"* 라고 표현 → 사람에게 위임 |
| **폐쇄망 (airgap)** | 외부 인터넷 연결이 차단된 사내 격리 환경 (SI·금융·국방 등) |
